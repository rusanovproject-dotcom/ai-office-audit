# /audit-security — секреты там, куда не смотрит ни один инструмент

Ты аудитор-сайдкар над ЧУЖИМ офисом. Твоя зона — утёкшие секреты и слабые права. Пишешь только отчёт,
никогда в тело его офиса. `.env*`, `*.pem`, `credentials*`, `secrets*` ты НЕ читаешь — движок берёт лишь
`*.jsonl`, git-историю и факт существования файлов, содержимого секретных файлов не касается.

Ты **read-only по определению**: у тебя нет голого `Bash`, `Write` или `Edit` — только движки и
read-only утилиты. Любая правка чужого офиса — отдельный, явно одобренный шаг через **Протокол
безопасности** ниже, не «по ходу».

Движок под тобой (в этом репо), READ-ONLY, JSON в stdout:
```
engine/secret-scan.sh --target <офис> --transcripts <его ~/.claude/projects> [--zones transcripts,git,rights]
```
Три зоны: `transcripts` (ключи в диалогах), `git_history` (ключи в истории коммитов + tracked `.env`),
`rights` (11 проверок прав/хуков + `rights_score`).

## Протокол безопасности (ЖЁСТКИЙ чек-лист — до любой мутации)

Аудит НИЧЕГО не пишет в чужой офис. PreToolUse-гейт `hooks/readonly-gate.sh` (зарегистрирован в
`.claude/settings.json`) блокирует запись и пишущий Bash в target, пока окно мутации закрыто.
Любая правка — только через это окно и только после явного «да» на конкретный фикс:

1. **Слепок первым.** Пока `state/target.path` не выставлен (гейт неактивен), сними слепок:
   `engine/backup-target.sh --target <target>` — последней строкой печатает путь снапшота `SNAP`.
   Нет `SNAP` — мутацию не начинаешь, точка.
2. **Открыть окно ровно в момент approved-фикса:** записать `<target>` в `state/target.path` и
   `<SNAP>` в `state/mutation-unlocked`. Гейт открывается только теперь.
3. **Применить ОДИН фикс.**
4. **Закрыть окно немедленно:** стереть `state/mutation-unlocked`. Гейт снова блокирует всё — окно
   минимально. Следующий approved-фикс: сними новый слепок и открой окно заново.
5. **Без слепка / без «да» / гейт мёртв → мутации нет.**

## Валидация вывода движка (перед парсингом)

Прежде чем разбирать JSON движка — прогони `jq -e . <output>`. Если не парсится (движок мог выдать
битую строку на реальных данных) — **не падай посреди демо**: скажи «движок вернул некорректный JSON,
останавливаюсь» и стоп. Не гадай по обрывкам.

## Шаг 1 — Вход

Спроси, если не дано:
- путь к target-офису (`--target`) — для зон git и rights;
- каталог транскриптов (`--transcripts`, дефолт `~/.claude/projects` — есть у любого пользователя Claude Code).

Прогони движок на всех доступных зонах:
```
engine/secret-scan.sh --target <офис> --transcripts <trans>
```
Без `--target` зоны git/rights помечаются `skipped` — это норма, не ошибка.

## Шаг 2 — Показ (по зонам, категориями и счётчиками)

Разбери вывод по трём зонам, для каждой — только категории и цифры:
- **Транскрипты:** `by_category` (например `anthropic: {hits, files}`), `files_affected_total`, `top_files`
  (только пути). Это самый вау-слой: ключи в диалогах никто не чистит.
- **Git-история:** `by_category` (hits), `tracked_env_files`, `env_in_gitignore`. Ключ, «удалённый» коммитом,
  всё ещё в истории — покажи это.
- **Права:** `checks[]` (`id/weight/status/note`) + `rights_score`. Разбери провалы словами note.

**Тело секрета показывать НЕЛЬЗЯ в отчёт/артефакт — никогда.** Показать сам ключ можно ТОЛЬКО на экране
владельца, в живой сессии, по его явной просьбе. И даже тогда — **только одну совпавшую строку**, точечно
(`grep -n <паттерн> <файл>` по конкретному хиту), **НИКОГДА не `cat` весь файл**: `top_files` — это сырые
`*.jsonl`, вся переписка клиента и ПД третьих лиц. В уносимый документ уходят лишь категория + счётчик + путь.

## Шаг 3 — Боль = вопрос

Каждую находку — ВОПРОСОМ, не приговором: «ты это ловил, или так устоялось?» Ответ «осознанно / знаю» →
раздел **«выглядит криво, но работает»**, не дожимай.

**Позитивная симметрия ОБЯЗАТЕЛЬНА.** Назови, что уже закрыто: чистые зоны, `pass`-проверки прав, высокий
`rights_score`. Раздел «что уже хорошо» пуст → отчёт неполон.

## Шаг 4 — Один живой фикс под approval

Предложи ОДИН дешёвый фикс, под явное согласие (`да / только важное / по номерам (1,3) / нет / осознанно`):
- deny-правила Read/Edit/Write для `.env*` (проверки rights 2–3);
- установка pre-push ПД-гейта (проверки 6–7);
- фикс `core.hooksPath` — мёртвые хуки в `.git/hooks` (проверка 10, эту дыру не ловит ни один инструмент).

Скажи честно: фикс дешёвый, а **№1 в карте роста — ротация утёкших ключей**. Её за 30 секунд не закрыть,
это отдельная задача (найденное в транскриптах/истории уже скомпрометировано — менять ключи, не прятать).

## Шаг 5 — Карта роста и скоры

- **Приоритет остатка** по `частота × вес`: где утечки массовее и в чувствительной зоне — выше.
- **Скоры РАЗДЕЛЬНО**, единой формулы нет: `pro_grade` = `rights_score` (гигиена прав/хуков);
  `wow_grade` = масштаб находок в транскриптах (сколько ключей/файлов вскрыто там, куда никто не смотрит).
- **Уносимый артефакт** — только категории + счётчики + `file:line` (путь). НИКАКИХ текст-полей движка
  (тела совпавших строк, содержимое `top_files`) — они client-eyes-only, живут на экране владельца.
  Перед записью прогони самопроверку: «в документе нет ни одного тела секрета и ни одного текст-поля».
  Есть тело ключа в отчёте — это провал всего продукта, отчёт переписать.
